The Maturity Roadmap
The Maturity roadmap reflects as one of the modules in the landing place and the overall page that navigates access to all the different modules in the system, but it is the core moule that explains and direct everything else. In fact, all other modules we designed could be explained as dependencies for this module. In this module we set the minimum performance standards that the organisation will comply with in the format of an audit tool. The principle is that we set the minimum performance criteria in a specific forma and structure, and we create an evidence upload or connect portal to evaluate the organisation compliance to this evidence in 5 levels (The Maturity levels). We then with Maturion aid and assistance including a human loop, evaluate that evidence and provide a life, continuous score of the operations maturity level. The tools that we designed thus far is there to help implement the evidence. For example, one of the criteria that an operational should comply with to ensure its proactive or resilient is Risk management. Hence the risk management tool that will create the evidence wee need to upload to ensure compliance with the operational maturity roadmap.

	Third and final level of approval. Once all Domains have been approved at level 2, the level 2 approver assigns the complete maturity audit roadmap, which by now has a name for example XYZ company Security Control Standard or XSCS to the highest level in the organisation for sign off. The approval loop is opened. The Third level approval is not happy, propose changes to the second level, the second level refers this to the first level for implementation. This approval loops undoes any previous lockups, and the user is able to submit the proposed changes, submitting back up into the approval loop, The loop remains open until final sign-off by the third level approval. IMPORTANT
The Audit model is configured and is like the building blocks of a house. It gets done in layers. Layer one is the domains and the domain cards. Layer 2 is the MPS’s. The Intent statements are part of layer 2. Layer three consists of the Assessment criteria. Layer 4 is the maturity level descriptors and level evaluator level, Level 5 is the evidence management level, and the last level is the human loop level, i.e., human auditor verification level. 3.	User registration and user fields
We already have a user field registration policy in the original app design. Basically, it involves a section where users are assigned roles and rights using a matrix. The principle we apply is this. The original user signs up after subscription, from thereon, forward, all users are invited. By a user already in the system. The main user will invite other organisations to join, and these will them become implementation users (I am not 100% how we named the user levels, please verify from original build). These “build users” will invite users at the domain level, users at the Domain level will invite users that will take responsibility at the MPS level. MPS level users will assign evidence managers for each domain. The user invite is structured such that the invite is accompanied by a radio checklist that invites users to become the owners of certain areas. If a Domain user invites people to take responsibility over MPS’s His invite modal with have a selector that lists all MPS’s for that domain. He can select one or more than one domain and when the user signs up he will automatically have read rights to the entire model but only have write rights to the ones assigned to him. This concept is relevant at all levels. As far as Level approvals are concerned, it works the same way. Approval always goes up one level above. If the approver is not yet a user, such as when the model is submitted for EXCO approval, The approver is also invited and his designation is entered. An approver that gets invited has only read rights, but dies have the approval workflow interface access, where he can reject and recommend changes.
