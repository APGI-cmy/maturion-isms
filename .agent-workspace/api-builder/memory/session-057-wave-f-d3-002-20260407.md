## Task Description

F-D3-002 Security Remediation: Remove the JWT Bearer authentication path from
`POST /api/ai/feedback/approve` (`api/ai/feedback/approve.ts`). The existing
`Authorization: Bearer` path accepted any structurally valid 3-part token without
signature verification, allowing any caller with `Bearer a.b.c` to trigger approve/reject
operations using `SUPABASE_SERVICE_ROLE_KEY`. Remediation: require `x-arc-token` exclusively.

## Lessons / What Future Sessions Should Know

1. **F-D3-002 pattern in pending.ts**: `api/ai/feedback/pending.ts` has the identical dual-auth vulnerability. It needs a separate remediation issue. The JWT payload extraction makes it more complex than approve.ts — needs a separate `organisationId` sourcing strategy before the Bearer path can be removed safely.

2. **pnpm not pre-installed in sandbox**: Need to `npm install -g pnpm@8.15.7` before running project tests. The project uses `workspace:*` protocol which requires pnpm.

3. **Test factory assertion for security tests**: When testing auth rejection, always assert the pipeline factory was NOT called (not just the status code). This satisfies FFA-04 security checks and ensures no service-role resources are accessed on failed auth.

4. **Wave scope**: This wave was tightly scoped — approve.ts only. Resist scope expansion for pending.ts without explicit CS2 authorization.
