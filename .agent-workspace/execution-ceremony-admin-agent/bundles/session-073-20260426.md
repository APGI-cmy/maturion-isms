## Suggestions for Improvement

**Concrete improvement — Scope Declaration Timing Convention (process gap):**

This session encountered the consistent pattern of `approved_artifact_paths[]` in the Foreman
scope declaration not listing ECAP bundle paths at time of ECAP appointment. This is structurally
inevitable: ECAP bundle paths cannot be known before ECAP is appointed, so they cannot be in a
scope declaration written before ECAP appointment.

**Recommendation**: Formalize a two-stage scope declaration protocol:
1. **Scope Declaration v1** (written by Foreman before builder appointment): lists all builder
   deliverable paths
2. **Scope Declaration v2** (written by Foreman after ECAP appointment but before IAA invocation):
   appends ECAP bundle paths + all ceremony artifact paths added by builder agents during Phase 4

A scope declaration template amendment to include a `pending_ecap_paths: []` field at v1 stage
would allow ECAP to confirm anticipated paths at appointment time without requiring the Foreman
to regenerate SCOPE_DECLARATION.md from scratch at step 2. This would reduce the risk of scope
parity issues (like the 14 vs 10 file discrepancy seen this session) going unnoticed until
immediately before IAA invocation.

**Impact**: Reduces pre-IAA cognitive load on Foreman; reduces scope-declaration churn; improves
traceability of ceremony artifact paths across the wave lifecycle.
