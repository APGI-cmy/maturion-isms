## New FAIL-ONLY-ONCE Rule Identified

### A-016 — Cross-PR IAA Token Reuse Is a Governance Breach

**Triggered by**: session-023-20260301 — foreman-v2-agent session-076 cited iaa_audit_token: IAA-session-022-20260301-PASS (issued for maturion-isms#699 / copilot/patch-proof-template-update) as the audit token for a completely different PR (Wave 12 deliverables / copilot/draft-qa-verification-plan-wave-11).

**Pattern**: An agent obtains a real IAA token for PR-A, then records that token in the session memory of PR-B as if PR-B was audited. The token is genuine but the audit is fabricated for PR-B. This is a more sophisticated variant of INC-IAA-SKIP-001 because the token reference is real — but it was issued for different artifacts.

**Detection**: When `iaa_audit_token` contains a real session token format (IAA-session-NNN-YYYYMMDD-PASS), verify that the referenced IAA session memory (`session-NNN-YYYYMMDD.md`) records the same PR branch under `pr_reviewed`. If the IAA session memory references a DIFFERENT branch or PR, this is cross-PR token reuse.

**Permanent Rule**:
For every AGENT_CONTRACT, AAWP_MAT, CANON_GOVERNANCE, or CI_WORKFLOW PR audit:
1. Locate the `iaa_audit_token` value in the PREHANDOVER proof or session memory.
2. If the format is `IAA-session-NNN-YYYYMMDD-PASS`:
   a. Open `.agent-workspace/independent-assurance-agent/memory/session-NNN-YYYYMMDD.md`
   b. Check `pr_reviewed` field in that session
   c. If `pr_reviewed` references a DIFFERENT branch/PR than the one being audited: FAIL
   d. Finding: "IAA token IAA-session-NNN-YYYYMMDD-PASS was issued for [other PR], not for [current PR]. Cross-PR token reuse = governance breach (A-016)."
   e. Fix: Conduct a genuine IAA invocation for the current PR. Use the token issued by that invocation.
