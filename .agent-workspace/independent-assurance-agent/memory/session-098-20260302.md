## fail_only_once_updates

**New pattern identified — A-023 CANDIDATE (ELEVATED to ACTIVE):**

The pattern of **pre-filling ceremony artifact fields with anticipated -PASS tokens before the ASSURANCE-TOKEN is actually issued** has now appeared across multiple sessions. In session-098, both:
- The PREHANDOVER proof checklist (line 111) was pre-filled with a completed token reference
- The foreman session-092 memory was fully populated as if session-097 had issued ASSURANCE-TOKEN

The anti-misuse comment in the PREHANDOVER template ("Never pre-fill '-PASS'") already exists, but the pattern persists because the Foreman writes ceremony files anticipating the outcome before it is issued. This is an A-017-class breach when the anticipated session actually issued REJECTION-PACKAGE.

**Recommendation**: Elevate the candidate A-023 rule: *"Ceremony artifacts must be committed with `iaa_audit_token: PENDING` and delegation log `result: [not yet issued — awaiting IAA session-NNN]`. Pre-filling token fields or `result` fields with anticipated -PASS values before ASSURANCE-TOKEN is received constitutes incorrect content (CORE-007) and potentially A-017. Populate token fields only AFTER receiving the actual ASSURANCE-TOKEN from IAA, via the Post-ASSURANCE-TOKEN ceremony."*

Adding A-023 to FAIL-ONLY-ONCE.md is flagged below under `fail_only_once_updates`.

## Suggestions for Improvement

**Suggestion S-017**: The PREHANDOVER template checklist section should be restructured so that the `IAA audit token recorded` item is explicitly marked `- [ ]` (UNCHECKED) with the PENDING placeholder text as the default. The current template gives an example of a completed item with a real token, which appears to have prompted the foreman to pre-fill it before the token was received. Changing the template to show the correct PENDING state (unchecked, with placeholder) would eliminate this error class. Low complexity; high impact given 2 consecutive REJECTION-PACKAGEs triggered by this exact issue.

**Suggestion S-018**: Session memory `iaa_audit_token` and delegation log `result` fields should contain explicit instructions in the template not to populate them until Post-ASSURANCE-TOKEN ceremony. Consider adding an inline comment: `# DO NOT POPULATE UNTIL IAA SESSION-NNN ISSUES ASSURANCE-TOKEN — Pre-fill = A-017 breach` adjacent to these fields in the session memory template.
