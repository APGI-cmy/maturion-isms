## Learning Notes

1. **SHA256 vs narrative accuracy**: Even when SHA256 checksums in a PREHANDOVER proof match the actual files, the narrative text descriptions can still be inaccurate. Future IAA invocations for AAWP_MAT PRs should explicitly cross-check version numbers, file names, and other factual claims in the narrative against actual file content. SHA256 integrity proves file content, but not that the description ABOUT that content is correct.

2. **HANDOVER_SUMMARY deliverable claims**: The HANDOVER_SUMMARY's "Files Modified" table is an auditable record. When it lists a file as CREATED, IAA must verify that file actually exists. This session found that session-038-20260302.md was listed as CREATED but is absent. This suggests governance ceremony sessions should have their own session memory creation verified before signing off on the HANDOVER_SUMMARY.

3. **Grafted commit tree false positives**: `git show --name-only` on a grafted (initial) commit shows ALL files in the repository tree, not just the changed files. Always use `git diff-tree --no-commit-id -r <SHA> --name-only` to get the actual changed files. This session correctly used diff-tree to resolve the false positive `.github/agents/` hit from `git show`.

4. **session-036 NOT_REQUIRED pattern**: When a session memory contains an incorrect `iaa_invocation_result: NOT_REQUIRED`, the correct remediation is to create a new governance ceremony session (like session-038) with PREHANDOVER proof and IAA invocation. The original session memory should NOT be retroactively edited. IAA should evaluate the NET state (is IAA being invoked?) rather than failing CORE-014 purely on historical record content that cannot be corrected without violating session memory integrity.

5. **Re-invocation convergence**: This is session-098, the second invocation for PR #805. Session-097 cited 11 failures. Session-038's corrective action resolved 9 of them clearly. Two governance documentation inaccuracies remain (version description error + missing session-038 memory file). The correction path is narrow and specific — PR #805 is very close to receiving ASSURANCE-TOKEN.
