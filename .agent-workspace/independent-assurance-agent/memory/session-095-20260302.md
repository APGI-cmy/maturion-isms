## Learning Notes

1. **O-001 mid-session fix pattern**: The producing agent applied a CS2 observation fix (O-001 directory guard) partway through the session. This changed the SHA256 of `agent-contract-audit.yml`. The agent updated the HANDOVER_SUMMARY with a placeholder `‚Äî` acknowledging the staleness, but did NOT update the session memory, and DID commit both artifacts with stale/incomplete hashes. This "mid-session fix without hash cascade" pattern should be flagged. It causes both CORE-007 and CORE-020 failures simultaneously.

   **Pattern name**: "Post-O-fix hash cascade miss" ‚Äî when a fix is applied after evidence artifacts are written, ALL hash references in ALL committed artifacts must be updated atomically before committing.

2. **OVL-CI-005 zero CI runs**: The PR had 0 CI check runs at the time of IAA invocation. This is the second time OVL-CI-005 has appeared as a failure (first was sessions 090-092 for PR #789). The PREHANDOVER template (in Foreman's memory space) does not enforce the `## CI Check Run Evidence` section for CI_WORKFLOW PRs. A-020 applies: the Foreman/governance-liaison PREHANDOVER template lacks the required CI_WORKFLOW overlay sections.

3. **OVL-CI-006 missing environment parity**: Simple to fix with a one-paragraph statement. The PREHANDOVER template should include this section by default for any CI_WORKFLOW PR.

4. **PREHANDOVER location in Foreman memory**: The governance-liaison-isms agent placed the PREHANDOVER in `.agent-workspace/foreman-v2/memory/` rather than its own workspace. This is not a failure per current checklist but is worth noting ‚Äî the governance-liaison-isms agent appears to be using the Foreman's PREHANDOVER template format. If the Foreman's template is stale (per A-020), the governance-liaison's PREHANDOVER will also be stale.

## learning_notes

1. **PHASE_A_ADVISORY claim pattern in liaison sessions**: The governance-liaison-isms agent session-036 wrote `iaa_invocation_result: NOT_REQUIRED; advisory mode PHASE_A_ADVISORY` in its session memory. The PREHANDOVER proof was correctly set to PENDING (the agent knew to invoke IAA), but the session memory was not updated to remove the NOT_REQUIRED claim. This is a session-memory-first-draft artifact ‚Äî the agent wrote session memory before IAA invocation and did not update the iaa_invocation_result field before committing. Fix pattern: agents should write `iaa_invocation_result: PENDING` in session memory at time of writing, and only update to the actual token post-ceremony. The session memory checklist item exists in the PR body ("Update session memory iaa_invocation_result with actual token") ‚Äî this is the correct post-ceremony flow, but the COMMITTED session memory must also say PENDING (not NOT_REQUIRED) before IAA invocation.

2. **OVL-KG-003 applies to newly created files**: Session-093 also saw a version history failure (on FAIL-ONLY-ONCE.md). The pattern is clear: new Tier 2 knowledge files frequently omit the ## Version History section. The fix is for agents to include it in their template when creating new knowledge files. This is a second occurrence of OVL-KG-003 failure on a newly created file. CANDIDATE pattern for strengthened guidance in the Foreman's prehandover-template.md.

3. **Category label divergence (AGENT_KNOWLEDGE vs KNOWLEDGE_GOVERNANCE)**: The KNOWLEDGE_GOVERNANCE category was added to IAA trigger table v2.1.0 on 2026-03-02 (same day as this session). The governance-liaison-isms agent used the label AGENT_KNOWLEDGE which may have been a working label during the period before v2.1.0 was published. The canonical name going forward is KNOWLEDGE_GOVERNANCE. Agents should load the current trigger table before classifying category in PREHANDOVER proofs. This appears to be a timing issue rather than a systematic failure.
