## Learning Notes

1. **Category mismatch from PREHANDOVER scope**: The invocation was requested as CI_WORKFLOW because PREHANDOVER-fix-ripple-agent-gate claimed `.github/workflows/ripple-integration.yml` in scope. However, the file was NOT changed in the PR. This pattern — where a PREHANDOVER proof describes work from a prior/different wave and is bundled with the actual PR — creates category confusion. Future invocations should always verify the PR diff via GitHub API before accepting the invocation's stated category.

2. **CANON_INVENTORY update is consistently missed**: This is the third or fourth invocation where CANON_INVENTORY hash staleness appears as a finding (CORE-006/OVL-CG-006). This is a systematic gap in the Foreman's pre-handover checklist. The pre-handover checklist should explicitly include "run sha256sum on every modified canon file and compare to CANON_INVENTORY — update if different."

3. **OVL-KG-002/003 pattern recurring**: index.md version bumps are also a recurring failure pattern (seen in session-092 for different branch, now again here). The Foreman's prehandover template should include a checklist item: "For every modified Tier 2 knowledge file: verify version was incremented and version history entry added in this commit."

4. **Dual-PREHANDOVER pattern**: This PR has two PREHANDOVER proofs for two different waves. The ripple-agent-gate proof describes a CI_WORKFLOW wave not represented in this PR. This is likely from a prior wave that was separated or merged differently. IAA should flag: when multiple PREHANDOVER proofs exist in one PR, verify each proof's scope aligns with actual PR diff.
