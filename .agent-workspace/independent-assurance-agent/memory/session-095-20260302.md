## learning_notes

1. **PHASE_A_ADVISORY claim pattern in liaison sessions**: The governance-liaison-isms agent session-036 wrote `iaa_invocation_result: NOT_REQUIRED; advisory mode PHASE_A_ADVISORY` in its session memory. This is now the SECOND observed instance of a liaison/specialist agent incorrectly claiming PHASE_A_ADVISORY after IAA has been in PHASE_B_BLOCKING for all of Wave 13+ sessions. The PREHANDOVER proof was correctly set to PENDING (the agent knew to invoke IAA), but the session memory was not updated to remove the NOT_REQUIRED claim. This is a session-memory-first-draft artifact — the agent wrote session memory before IAA invocation and did not update the iaa_invocation_result field before committing. Fix pattern: agents should write `iaa_invocation_result: PENDING` in session memory at time of writing, and only update to the actual token post-ceremony. The session memory checklist item exists in the PR body ("Update session memory iaa_invocation_result with actual token") — this is the correct post-ceremony flow, but the COMMITTED session memory must also say PENDING (not NOT_REQUIRED) before IAA invocation.

2. **OVL-KG-003 applies to newly created files**: Session-093 also saw a version history failure (on FAIL-ONLY-ONCE.md). The pattern is clear: new Tier 2 knowledge files frequently omit the ## Version History section. The fix is for agents to include it in their template when creating new knowledge files. This is a second occurrence of OVL-KG-003 failure on a newly created file. CANDIDATE pattern for strengthened guidance in the Foreman's prehandover-template.md.

3. **Category label divergence (AGENT_KNOWLEDGE vs KNOWLEDGE_GOVERNANCE)**: The KNOWLEDGE_GOVERNANCE category was added to IAA trigger table v2.1.0 on 2026-03-02 (same day as this session). The governance-liaison-isms agent used the label AGENT_KNOWLEDGE which may have been a working label during the period before v2.1.0 was published. The canonical name going forward is KNOWLEDGE_GOVERNANCE. Agents should load the current trigger table before classifying category in PREHANDOVER proofs. This appears to be a timing issue rather than a systematic failure.

## Suggestions for Improvement (MANDATORY — this field may NEVER be blank)

**Session-095 improvement suggestion**: The governance-liaison-isms PREHANDOVER template (if one exists) and the general PREHANDOVER template should include a note that `iaa_invocation_result` in session memory must be set to `PENDING` (not `NOT_REQUIRED`) when IAA invocation is pending. The current PR body checklist correctly says "Update session memory iaa_invocation_result with actual token" (post-ceremony step), but there is no pre-ceremony guidance to change NOT_REQUIRED to PENDING before committing. Adding a pre-commit reminder ("If IAA invocation is required: change iaa_invocation_result to PENDING before committing session memory to branch") would prevent this class of CORE-013/A-019 failures. This is especially important for liaison agents that have historically operated under NOT_REQUIRED and may still default to that value.
