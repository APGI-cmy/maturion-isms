## Learning Notes

1. **Dual-category PRs require both overlays**: This PR is both CANON_GOVERNANCE (canon file + CANON_INVENTORY) and KNOWLEDGE_GOVERNANCE (Tier 2 files). The IAA correctly applied both overlays. Future reminder: when a PR touches canon files AND Tier 2 knowledge files, both CANON_GOVERNANCE and KNOWLEDGE_GOVERNANCE overlays must be loaded and applied in full.

2. **OVL-CG-004 and OVL-CG-005 are consistently at risk**: These two checks (ripple assessment and drift evidence) are the most commonly missed CANON_GOVERNANCE overlay checks. The PREHANDOVER proof template should be enhanced to include explicit prompts for these items. Suggestion: add "## Ripple Assessment" and "## Drift Evidence" sections to the PREHANDOVER template for CANON_GOVERNANCE PRs.

3. **OVL-KG-003 (version history) commonly missed for index.md**: The index.md file did not have a version history table at all — this was likely a pre-existing omission that became a finding when the file was first modified under the v2.2.0 KNOWLEDGE_GOVERNANCE overlay. The PREHANDOVER template or index.md template should include a version history section by default.

4. **Content quality vs. process quality**: This is the third instance where substantively correct content (the zero-severity-tolerance policy is well-implemented) was accompanied by process gaps in the PREHANDOVER proof. The content of what was implemented is sound; the evidence bundle packaging is what triggered rejection. This is a recurring pattern — see also session-085 (KNOWLEDGE_GOVERNANCE ceremony introduced because process was missing) and session-086 (liaison admin records missing evidence). Consider a PREHANDOVER proof checklist specifically targeting OVL-CG-004/005 and OVL-KG-003 as high-risk items for agents to self-verify before submitting.

5. **Zero-severity-tolerance self-validates**: This session is the first invocation where CORE-021 (Zero-Severity-Tolerance) was active at the time of audit. The rule correctly prevented any "minor" characterisation of the 3 findings — all 3 produce REJECTION-PACKAGE. The rule works as designed.

2. **index.md structural requirement**: The Tier 2 knowledge index.md should carry a version history table as a mandatory structural element (like every other Tier 2 file already does). The absence of this table in index.md is a template deficiency. Consider updating the session memory template or Tier 2 index template to always include a version history section.

The "before hash" for (a) is not always easily obtainable from the producing agent (requires looking up the prior committed state).
