## Learning Notes

1. **A-032 Schema Compliance — CHECK Constraint Values**: This session confirms that A-032 must be extended to cover not just column NAMES but also CHECK CONSTRAINT VALUES for constrained columns (capability, event_type, etc.). Column name compliance is necessary but not sufficient — the values that flow through must also comply with the schema constraints.

2. **In-Memory → Supabase Transitions**: When a PR converts an in-memory implementation to a Supabase-backed one, IAA must treat it as a full schema compliance audit for the target table. Pre-existing schema drift that was harmless in-memory becomes a real runtime failure after the wiring. F-001 in this session is the canonical example.

3. **PREHANDOVER SB Attestations**: When a PREHANDOVER proof includes scope blocker compliance attestations (SB-00X), IAA should directly verify each attestation against the actual schema/code, not accept them at face value. The SB-002 attestation "values match CHECK constraint" was accepted by the builder but was incorrect.

4. **MemoryLifecycle fire-and-forget pattern**: The `.catch(() => {})` pattern in MemoryLifecycle silently swallows episodic recording errors. This pattern is a known risk multiplier — schema mismatches, RLS failures, and type errors are all silently absorbed. When reviewing any PR that writes to episodic memory, IAA should trace the call chain through MemoryLifecycle to confirm the catch wrapper doesn't mask real failures.

5. **Positive observation**: The implementation quality of EpisodicMemoryAdapter.ts is excellent. The SupabaseMinimalClient interface pattern, toRow/fromRow helpers, RLS enforcement via rpc('set_config') before INSERT, error surfacing, and test coverage are all first-class. The single finding (F-001) is a schema alignment issue, not a code quality issue. Expected re-audit to result in ASSURANCE-TOKEN once the migration is added.

## Suggestions for Improvement (MANDATORY)

1. **IAA Knowledge Enhancement — A-032 Extension**: Add a note to FAIL-ONLY-ONCE A-032 explicitly requiring IAA to cross-check not just column names but also CHECK CONSTRAINT VALUES for any constrained column in an INSERT. The current A-032 text focuses on column names; value compliance is equally important. Applicable future sessions: any PR with INSERT into tables that have CHECK constraints on non-id columns (capability, event_type, status columns, etc.).

2. **Pre-Brief Enhancement**: Future Pre-Briefs for AAWP_MAT PRs involving type-to-DB-column mappings should explicitly call out CHECK constraint values in the SB items. The SB-002 section in this wave's pre-brief listed the 8 CHECK values but did not cross-check against the TypeScript enum. Adding an explicit IAA pre-brief note: "Verify: TypeScript type enum values ⊆ DB CHECK constraint values" would catch this class of issue earlier.

3. **Parking Station**: The MemoryLifecycle fire-and-forget `.catch(() => {})` pattern should be assessed at the next wave for whether important failure modes (not just RAG, but any future schema drift or RLS violation) should surface to callers or monitoring. This is an architectural observation for FM review, not a REJECTION-PACKAGE finding.
