## Learning Notes

**Key learning this session**: The PREHANDOVER-and-session-memory-uncommitted pattern is recurring. A-033 was codified for exactly this reason (INC-CI-GATEWAY-FIX-001-IAA). The pattern seems to occur when producing agents complete the governance ceremony but omit the final `git add + git commit` step for memory/PREHANDOVER files. In this session, the Foreman's PREHANDOVER artifact table correctly showed `✅` (without a commit SHA) for the PREHANDOVER and session memory files — indicating the files were prepared but not committed. IAA correctly identified this via git verification and issued REJECTION-PACKAGE.

**Process observation**: The PREHANDOVER table should distinguish between "file committed" (with SHA) and "file prepared/pending commit" (without SHA). The absence of a SHA in the artifacts table is a reliable signal that the file may be uncommitted. IAA should treat a missing SHA in the PREHANDOVER artifacts table as a pre-warning for the A-033 git check.

**Note on trivial fix**: The content of the wave is entirely correct. All substantive, content, and overlay checks PASS. The REJECTION-PACKAGE is solely due to the governance ceremony completion step (git commit) being missed. The fix is one commit. ## Suggestions for Improvement

**Session suggestion**: The PREHANDOVER proof artifacts table should use a formal "COMMITTED" vs "PENDING_COMMIT" status rather than ambiguous `✅` symbols. Adding a required commit SHA column (with "PENDING" allowed for the PREHANDOVER itself and session memory, since those are committed after the IAA token) would clarify the ceremony state. Suggestion: Foreman v2 contract should require a `status: COMMITTED | PENDING_COMMIT` field in the PREHANDOVER artifacts table for each artifact.
