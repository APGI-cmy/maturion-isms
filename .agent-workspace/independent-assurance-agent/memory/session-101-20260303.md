## Learning Notes

**learning_notes**:

1. **A-021 pattern recurrence (third occurrence)**: Session-040 exhibits the same pattern as sessions 090 and 091 (PR #789) — the producing agent performed extensive remediation work in the working tree but invoked IAA before committing any of it. A-021 was codified precisely for this pattern. The governance-liaison-isms-agent FAIL-ONLY-ONCE.md (v1.1.0 at time of session-040) does NOT yet include A-021 equivalent. Recommendation: governance-liaison-isms-agent should add an equivalent of A-021 to their own FAIL-ONLY-ONCE.md.

2. **CORE-017 structural issue — authorized-revert by unauthorized agent**: The session-040 remediation attempts to revert a .github/agents/ file by the same unauthorized agent class that made the breach. Even a "revert" is still a modification of a protected file. The governance-liaison-isms-agent cannot self-heal a .github/agents/ breach by making another .github/agents/ modification. The correct path is: (a) commit non-agent-file remediation artifacts, (b) create escalation for CS2/CodexAdvisor-agent to perform authorized revert in separate PR, (c) invoke IAA for the non-agent-file portion only. This is a new pattern — documenting for FAIL-ONLY-ONCE consideration.

3. **PREHANDOVER ceremony order enforcement**: This session confirms that even in breach-remediation contexts (where there is urgency to fix), the PREHANDOVER ceremony is not bypassed. PREHANDOVER proof → commit → invoke IAA is the invariant sequence regardless of context. The urgency of a breach does not override the ceremony.

4. **CS2 co-authorship != CodexAdvisor-agent producing**: The session-039 commit was co-authored by APGI-cmy (CS2). This satisfies the CS2 authorization condition of CORE-017 but NOT the CodexAdvisor-agent producing condition. CORE-017 requires BOTH conditions. Future auditors should not conflate CS2 co-authorship with CodexAdvisor-agent identity. Note this pattern for possible FAIL-ONLY-ONCE addition.
