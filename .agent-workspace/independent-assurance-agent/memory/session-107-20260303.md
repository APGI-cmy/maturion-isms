## Learning Notes

1. **Three-invocation resolution pattern**: This PR required 3 IAA invocations to reach ASSURANCE-TOKEN. Session-105 caught 7 failures (primarily commit-before-IAA and wrong issue reference); session-106 caught 1 residual failure (OVL-CI-005 URL not in committed file despite being available); session-107 issued ASSURANCE-TOKEN. The pattern is: (1) first invocation catches procedural failures at scale; (2) second invocation catches partial-fix failures (fix done externally but not committed); (3) third invocation passes when all fixes are properly committed. A-021 and the "claimed-resolved vs. committed-artifact discrepancy" pattern (session-106 learning note 1) remain the most common sources of multi-invocation PRs.

2. **OVL-CI-005 for workflow_dispatch-only workflows — resolved pragmatically**: The polc-boundary-gate CI run URL (22618191766) was accepted as satisfying OVL-CI-005 for ripple-integration.yml (a workflow_dispatch-only workflow). This establishes a precedent: for workflows that cannot auto-run on push, a URL from ANY CI check that ran on the same branch as a direct result of the push is acceptable evidence. YAML validation + code review (OVL-CI-004) provides complementary assurance on the modified workflow's correctness. CS2 should be invited to formally codify this interpretation if they wish (session-105 S-002 suggestion remains open).

3. **Session-106 claimed-resolved verification pattern working**: The session-106 suggestion (verify committed file content before re-invoking IAA) was followed this session — the foreman ran `git show HEAD:...PREHANDOVER... | grep "actions/runs"` and confirmed the URL was in the committed file before invoking IAA (per the invocation summary provided by CS2). This pattern should be added to the foreman's FAIL-ONLY-ONCE.md as a standing pre-IAA verification step.

4. **Substantive work quality**: ripple-integration.yml awk parser and escalation hardening are technically sound. OVL-CI-004 confirms policy correctness. Foreman PREHANDOVER template v1.2.0 with OVL-CI-005/006 sections will prevent cascading REJECTION-PACKAGEs on future CI_WORKFLOW waves. The governance value delivered by this PR is high.
