## learning_notes

1. **Pre-existing TODO propagation across REJECTION cycles**: When a TODO is present in a delivered artifact from the initial delivery commit, it can propagate across multiple REJECTION-PACKAGE cycles undetected if each REJECTION session only re-checks the specific files that were the subject of that session's findings. This session found a TODO in EmbeddedAIAssistant.tsx that was present since commit `de8472d` — through all three handover attempts. Session-141-v2 checked CORE-007 for DashboardPage.tsx but not for all other modified files.

2. **PREHANDOVER zero-TODO claims require comprehensive grep evidence**: The PREHANDOVER v3 claimed "Zero TODO keywords in production code ✅" without grep evidence. This claim was incorrect. Future PREHANDOVER proofs should include the actual grep command output (or a reference to it) as evidence for this claim.

3. **One root finding, multiple check failures**: The `organisationId: 'default'` with TODO comment drives CORE-007, BD-002, AND BD-003 failures simultaneously. The three failures share a single root cause and a single fix. This is a legitimate multi-check failure from one root finding — not severity inflation.

4. **Tests pass but functionality is broken**: 40/40 tests pass despite the TODO being present. This is because the test suite tests structural/interface correctness, not runtime org context wiring. Test coverage for the org isolation aspect of the AI assistant is a Batch C concern.

## Suggestions for Improvement (MANDATORY)

1. **Comprehensive CORE-007 grep in PREHANDOVER template**: The PREHANDOVER template should include a mandatory CORE-007 pre-check step: `grep -rn "TODO:\|FIXME:\|STUB" $(cat SCOPE_DECLARATION.md | grep -E "^\- \`" | sed "s/.*\`\(.*\)\`.*/\1/")` — run against all files declared in SCOPE_DECLARATION and include the output in the PREHANDOVER proof as evidence. This would have caught the EmbeddedAIAssistant.tsx TODO in the original delivery.

2. **SCOPE_DECLARATION file listing with TODO grep output**: Every PREHANDOVER ceremony should include the actual grep output proving zero-TODO compliance, not just a checkbox claim. The producing agent should paste `grep -rn "TODO\|FIXME\|STUB" <all-modified-files> || echo "CLEAN"` output directly into the PREHANDOVER proof.
