## Learning Notes

1. **Single-fix resolution pattern is efficient**: When a REJECTION-PACKAGE cites exactly one failure (OVL-AC-011 stale char counts), and the producing agent commits only the targeted fix without touching other files, re-verification is fast and unambiguous. The commit scope (PREHANDOVER-only) made this the cleanest re-invocation of the four-session sequence.

2. **Drift table requires dual verification**: OVL-AC-011 requires both char counts AND git object hashes to be verified independently. The git object hash verification (`git rev-parse HEAD:<path>`) is a reliable authoritative check that cannot be fooled by stale cached values. The session-103 failure was exclusively in the char count column, while hashes were already accurate. This asymmetry is a useful pattern to note for future drift table audits.

3. **Four-invocation resolution sequence**: sessions 101→102→103→104 demonstrate the STOP-AND-FIX mechanism working as designed. 8 failures → 4 new failures → 1 new failure → 0 failures. Each session's REJECTION-PACKAGE was precise and actionable, enabling targeted fixes. The incremental nature (new failures surfaced after old ones fixed) is expected behavior, not a governance gap.
