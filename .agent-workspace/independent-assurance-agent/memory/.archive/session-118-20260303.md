## Learning Notes

1. **A-021 + CORE-018 recurrence**: PREHANDOVER and session memory left as untracked files is a persistent failure pattern. The issue is workflow order: files are written to disk, then the implementation commit is made, but the memory files are not staged with the commit. Pre-commit hook or explicit Foreman workflow step needed.

2. **SCOPE_DECLARATION staleness (A-026)**: This wave's SCOPE_DECLARATION.md was carried from the previous branch commit (Initial plan) which pulled in the Wave 13 scope declaration. When a new wave is started on a branch that already has a prior SCOPE_DECLARATION.md, foreman must update it as the first committed file, not the last.

3. **Knowledge file version header neglect**: The FAIL-ONLY-ONCE.md header field was at v2.2.0 while the footer was at v2.5.0 — three version cycles of neglect. The version footer and version history table exist but the header `**Version**:` field was not touched. This suggests foreman's knowledge update procedure only updates the footer and table, not the header. All three locations must be updated atomically.

4. **PREHANDOVER ceremony sections are not scaffolded**: OVL-AM-004, -005, -006 and OVL-KG-001 through -005 failures all result from absent sections in the PREHANDOVER proof. If the PREHANDOVER template pre-populates these as required sections (even with placeholder text that must be replaced), forgetting them becomes structurally impossible. This is the highest-leverage improvement available.
