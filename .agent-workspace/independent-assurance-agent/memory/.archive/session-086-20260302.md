1. **Governance-liaison ripple sessions have own PREHANDOVER proof format gap**: This session is the first IAA audit of a governance-liaison ripple session in Phase B. The governance-liaison contract correctly lists `prehandover_proof` as a mandatory IAA artifact. However, the governance-liaison has not previously created a dedicated PREHANDOVER proof file for ripple sessions — the HANDOVER_SUMMARY.md has served as the handover evidence. In Phase B, a dedicated PREHANDOVER_PROOF.md with `iaa_audit_token` and `## IAA Agent Response (verbatim)` section is required. Future ripple sessions should create this file as part of the session workflow (before invoking IAA).

2. **Category classification for governance admin PRs**: The AMBIGUOUS classification is correct for governance-liaison ripple PRs where new governance tracking files are created. The trigger table should potentially add a "GOVERNANCE_ADMIN" category with explicit handling (either EXEMPT for pure no-drift admin records, or AMBIGUOUS with mandatory IAA). This would reduce classification uncertainty for future ripple sessions.

3. **Stale advisory_phase in governance-liaison contract**: The `advisory_phase: PHASE_A_ADVISORY` in the governance-liaison contract is stale (IAA now PHASE_B_BLOCKING). This should be updated via the normal CodexAdvisor + IAA + CS2 channel. Flagged to CS2 as advisory.

4. **All substantive work verified correct**: When re-auditing after the PREHANDOVER proof is created, the substantive checks (AUDIT-1 through AUDIT-6) should be quick — the underlying work is confirmed correct. The re-audit can focus on ceremony compliance.
