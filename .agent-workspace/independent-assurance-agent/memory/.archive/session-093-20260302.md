## Learning Notes

1. **Commit cae9a53 fixed OVL-KG-004 (session-092 failure) but introduced OVL-KG-002 (new failure)**: When committing the fix for a stale index.md, the Foreman/builder added rules A-021/A-022 to the IAA FAIL-ONLY-ONCE.md and added a version history entry for 1.5.0 — but did NOT update the file's `**Version**:` header field. The version history and index were updated correctly; only the header was missed. This is a pattern: multi-file governance commits that touch version history entries may miss updating the file's own version header. **CANDIDATE for FAIL-ONLY-ONCE rule**: "When adding rules to a Tier 2 FAIL-ONLY-ONCE file and adding a version history entry, ALWAYS update the `**Version**:` header field to match. Header + history must be in sync. Updating only the history without the header = OVL-KG-002 failure."

2. **Six invocations for a single PR is becoming a pattern**: This is the fifth invocation for PR #789 and a sixth (session-094) will be needed. Each REJECTION-PACKAGE has been for a different, incremental governance gap discovered in the fix for the prior gap. Contributing factor: when remediating one OVL-KG check, the Foreman's commit simultaneously introduces another OVL-KG failure. This suggests the PREHANDOVER pre-commit checklist needs an explicit item: "For each Tier 2 knowledge file you modified: verify `**Version**:` header matches the version history entry you added. Count the lines you changed. If you added a version history row, confirm the file's header was also updated."

3. **OVL-KG-002 detection timing**: The check for this failure is straightforward — grep the `**Version**:` header and compare to the version history table and index. Future sessions can add this as a fast pre-check: `grep "^\*\*Version\*\*:" <file>` and compare to what the index claims.

4. **CANON_INVENTORY parsing key**: The CANON_INVENTORY.json uses key `canons` (not `files`) for the list of entries. Python parsing should use `data.get('canons', data.get('files', []))` to handle both formats. This prevented correct hash verification in early Phase 1 execution (corrected mid-session). Future sessions: use the correct key.
