**Pattern observation**: The absence of ceremony artifacts (PREHANDOVER proof + session memory) for a CS2-direct PR is a new pattern not previously encountered. This session reveals that the requirement applies equally to CS2-direct copilot-swe-agent PRs — the ceremony artifacts must be created even when CS2 creates the PR directly. This is the most extreme SCOPE_DECLARATION staleness observed: the content is from a different PR entirely, not just missing a few recently-added files. The fix is straightforward but requires resetting SCOPE_DECLARATION.md to match this PR's actual diff.

**Security finding (NEW pattern — OVL-CI-004 script injection)**: First session where a workflow script injection vulnerability was identified. The pattern: `${{ steps.X.outputs.Y }}` embedded directly in a github-script `script:` block (as opposed to using `env:` variables). This is a well-known GitHub Actions security anti-pattern (CWE-1336, script injection via expression). The mitigation is always to use `env:` variables for step output values that may contain attacker-influenced content. Flagging for FAIL-ONLY-ONCE consideration.

## Learning Notes

1. **CS2-direct PRs require ceremony artifacts too**: This is the first IAA session where the producing agent is CS2/copilot-swe-agent directly (not foreman-v2 through the normal governance chain). The CORE-018 ceremony requirements (PREHANDOVER proof + session memory + iaa_audit_token) apply regardless of who created the PR. The foreman-v2 knowledge index governance ceremony gate explicitly lists `.github/workflows/**` and `.agent-workspace/foreman-v2/knowledge/*.md` as governed paths requiring full ceremony. CS2 invocations of copilot-swe-agent must include ceremony artifact creation in the task scope.

2. **Workflow script injection pattern to add to FAIL-ONLY-ONCE**: The pattern `const X = \`${{ steps.N.outputs.Y }}\`` in github-script steps is a script injection anti-pattern. All IAA CI_WORKFLOW overlay checks should include a scan for this pattern. The correct pattern is `env: X: ${{ steps.N.outputs.Y }}` and `const x = process.env.X`. This applies to ALL future workflow reviews.

3. **SCOPE_DECLARATION.md re-anchor window**: When a PR branch is forked off a branch that already had a SCOPE_DECLARATION.md (for different work), the new PR inherits the old SCOPE_DECLARATION.md. The A-026 rule catches this, but the pattern is worth noting: branches created from feature branches (not main) carry forward the source branch's SCOPE_DECLARATION.md.

4. **Pre-fixes quality**: The two pre-fixes in commit 02f1217 (version ref 1.10.0→1.7.0 and severity contradiction) were correctly applied. The WAVE-CURRENT-TASKS-PROTOCOL.md is logically consistent and complete. The knowledge document substance is good. OVL-KG-002/003/004/005 all pass. The REJECTION-PACKAGE is entirely about ceremony artifacts and one security flaw — not about substantive content quality.

5. **github-script injection scan scope**: Future CI_WORKFLOW overlay OVL-CI-004 checks should include a regex scan of the workflow file for the pattern `` `${{ `` inside `script:` blocks of `github-script` steps. This is an IAA-level enforcement point.

## Suggestions for Improvement (MANDATORY)

1. **New FAIL-ONLY-ONCE rule candidate — GitHub Actions script injection**: The `const x = \`${{ steps.Y.outputs.Z }}\`` pattern in github-script steps is a script injection vulnerability. IAA should codify a rule (e.g., A-027) that any `.github/workflows/*.yml` file containing `${{ ` inside a `script:` block of `actions/github-script` must use the `env:` variable pattern instead. This is a standing security check for all future CI_WORKFLOW overlay reviews.
