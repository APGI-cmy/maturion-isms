## Learning Notes

1. **Character limit boundary sensitivity**: governance-liaison-isms-agent.md was at 29,993 chars — only 7 chars under the 30,000-char limit. The +8-char rename pushed it to 30,001. For bulk renames that add characters to ALL files, CodexAdvisor must pre-check character counts against the limit BEFORE applying the rename to files that are within 8 chars of the boundary. A "chars to limit" column in the drift table would prevent this.

2. **CodexAdvisor-agent.md size: first-time OVL-AC-009 enforcement**: CodexAdvisor-agent.md is 36,184 chars — significantly over 30,000. This is the first session where IAA has enforced OVL-AC-009 against CodexAdvisor-agent.md (prior agent contract PRs audited by IAA did not include CodexAdvisor.md). CS2 waiver or significant refactoring required. This is likely to recur in future sessions unless CS2 grants a waiver or the contract is restructured.

3. **OVL-AC-011 delta arithmetic**: PREHANDOVER stated "+7 chars per file" but actual measured delta is +8 per file. The discrepancy: "secret:" is 7 chars (no trailing space), but in the YAML file the token is `secret: ` (8 chars including the space before the value). CodexAdvisor miscounted by omitting the space. Future PREHANDOVER proofs must measure the delta from the actual file bytes, not by counting field name length without context characters.

4. **Aggregate vs per-file drift tracking**: OVL-AC-011 requires per-file drift data. For bulk renames, it is tempting to use a single aggregate row — but this fails the check. Even when all 16 files have the same delta, OVL-AC-011 requires 16 individual rows with per-file before/after counts. CodexAdvisor should be instructed to always produce per-file drift tables for bulk changes.

5. **Self-consistency in PREHANDOVER claims**: The PREHANDOVER stated "±0 chars net change" AND "+7 chars per file" in the same cell — mutually exclusive claims. Future PREHANDOVER proofs should be self-consistent: pick one statement and verify it against measured values.
